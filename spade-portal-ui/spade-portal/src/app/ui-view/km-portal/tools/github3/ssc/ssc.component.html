Automatic Publishing of Software Assets

Software assets on Github3 are automatically published onto Service Software Catalog provided they are part of Standard folder structure described below. For more details on Branching and Structure refer GitHub3 Branching & Structure

GitHub3 standard folder structure
    Community : AS Community
Repositories under AS-Community
  • ACI
  • NSO
  • UCSD
  • VMS
  • Analytics
  • XSpeed

Each repository maps to a Domain in Service Software Catalog

Each repository has the following folder structure
    Repository (Domain) –> Subfolder (Categories) –> Software Assets –> Asset Contents

Each folder maps to a Category in the Service Software Catalog
Under these folders there can be multiple sub folder each corresponding to a software asset.

Each software asset should have a Readme.md file with the following format.
    Sample Readme.md
      # Description

      Best Practice Linter for ACI checks for a number of AS best practice configurations and reports if those are not present or
      misconfigured.

      # Overview

      The script contains links to various plugins which are invoked in a serial fashion and
      the criterion in each plugin is verified. Should there be a misconfiguration or BP
      violation, the script outputs the object and the misconfiguration and suggests the user to
      correct the same.

      The plugins supported currently are for the following checks:
      - Version of APIC code running.
      - Check if IP learning on BD is limited to the subnet configured on the BD.
      - Check if ARP flooding is enabled in the BD.
      - Check if GARP learning is enabled on the BD.

      # Author

      krchaita

      # Version

      1.0

      # Release Date

      22/08/2016

      # Usage

      python bplint.py -uri https://apic -username admin -password ins3965!

Whenever a new asset is added or or an existing asset is updated in GitHub3 under AS-Community within the standard folder structure described above, the changes will be reflected automatically within few minutes.

A webhook has been defined in git which gets triggered whenever there is any change (Add/Update/Remove) within the AS-Community. Webhook notifies the list of software assets changed to Service Software Catalog. For the changed assets latest Readme.md is read and contents are updated in the Service Software Catalog.

Steps to Publish the Software assets on Service Software Catalog from GitHub3
  • Use case 1 -  New software asset is added within the standard folder structure with standard Readme.md

    • When a new software asset is put into GitHub3 the details are automatically pulled into Service software Catalog and published.
    • Whenever the new asset is successfully published a mail will be sent to Architect, Author, Product owner and Contributors and to the Admin
    • If the Readme.md is missing then the Asset will not be published and a notification mail would be sent to the submitter and Admin
    • Architect, Author, Product owner and Contributors will have an edit option for the asset in the Service Software Catalog. Edit option should be used to
      • Add an Icon for the asset. Asset gets added with default icon for the first time when it is automatically pulled from Github3.
      • Add the key words. Based on keywords software assets are listed when the search is done on Service software catalog
      • If Software asset needs to be associated with other Domains and Catagories please contact the Admin

  • Use case 2 -  New software asset is added outside the standard folder structure with standard Readme.md

    • When a new software asset is put into GitHub3 outside the standard folder structure the details are NOT automatically pulled into Service software Catalog. Corresponding new domain and/or category needs to be added to Service Software Catalog by the Admin.
    • Send a mail with details to Admin . Admin will add the domain and/or category and will confirm. Then follow the steps in Use Case 1 to get it published.

  • Use case 3 -  Anytime Existing Software Asset within the standard folder structure is updated

    • When any software asset is updated in GitHub3 corresponding Readme.md is read by Service Software Catalog and its contents are updated.
    • If the changed contents of the Readme.md are not getting reflected in Service Software Catalog please contact the Admin with details like time and asset that got updated.

  • Use case 4 -  Removing the software asset from the Service Software Catalog

    • Architect, Author, Product owner and Contributors have an edit option for the asset in the Service Software Catalog. Edit option has a check box to hide the asset. When checked the Software asset does not show up in search and listing.
    • If the Software Asset needs to be permanently removed then first remove it from GitHub3. Then hide the Asset in Service Software Catalog. Then please contact the Admin with Asset name. Currently adding/updating assets is automated; Removal can only be done by the Admin.
    • Please note that if the asset is removed from Service Soaftware Catalog but is still present in GitHub3 then it will be added back next time whenever there is change in GitHub3 in that asset folder.
